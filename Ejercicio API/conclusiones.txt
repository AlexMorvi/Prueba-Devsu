===============================================================================
                    CONCLUSIONES DEL EJERCICIO API PETSTORE
===============================================================================

Proyecto: Automatización de Pruebas API REST para Swagger Petstore
Framework: Karate DSL
Tecnologías: Java, Maven
Arquitectura: Behavior Driven Development (BDD), Patrón Create-First

===============================================================================
                                RESUMEN EJECUTIVO
===============================================================================

Se implementaron exitosamente las pruebas automatizadas para la API de Petstore 
utilizando Karate Framework. El proyecto cubre el ciclo de vida completo (CRUD) 
de la entidad "Pet", garantizando la integridad funcional de los endpoints 
principales mediante escenarios independientes y autocontenidos.

### CASOS DE PRUEBA IMPLEMENTADOS

1. POST /pet (Añadir mascota) [@addPet]
   - Generación dinámica de payload con ID único basado en timestamp.
   - Verificación de creación exitosa y validación de contrato.

2. GET /pet/{petId} (Consultar mascota) [@getPetById]
   - Implementación del patrón Create-First para evitar falsos negativos por falta de datos.
   - Consulta y validación estructural de la respuesta HTTP 200.

3. PUT /pet (Actualizar mascota) [@updatePet]
   - Modificación de atributos estáticos (cambio de estado a "sold").
   - Confirmación de persistencia de los cambios aplicados en la base de datos de la API.

4. GET /pet/findByStatus (Búsqueda por estado) [@findPetsByStatus]
   - Filtrado por estado con validación flexible de esquemas JSON.
   - Comprobación de consistencia en el manejo de arreglos de objetos.

### RESULTADO FINAL
La totalidad de los escenarios fueron ejecutados exitosamente, asegurando una 
evaluación estricta de variables de entrada, códigos de retorno y esquemas de respuesta.

===============================================================================
                                IMPLEMENTACIÓN TÉCNICA
===============================================================================

1. ESTRUCTURA Y MANTENIBILIDAD
   - Separación de esquemas JSON para validaciones rígidas y flexibles (strict/flex).
   - Centralización de variables dinámicas vía karate-config.js.
   - Uso de custom tags para habilitar ejecuciones atómicas por endpoint.

2. ESTRATEGIA DE DATOS
   - Generación de identificadores inmutables estocásticos para prevenir colisiones 
     entre hilos de ejecución paralelos.
   - Aplicación intensiva del bloque "Background" y abstracción de datos para inyectar 
     estados limpios antes de cada llamada de red.

===============================================================================
                            HALLAZGOS TÉCNICOS PRINCIPALES
===============================================================================

1. FUNCIONALIDAD DE LA API
   - Tiempos de latencia estandarizados y acordes a los esquemas funcionales previstos.
   - Respuestas predecibles controladas frente a inyecciones de datos no contemplados (ej. HTTP 404).

2. LIMITACIONES Y ALCANCES DE SEGURIDAD EN EL DISEÑO
   - Naturaleza de Sandbox (pública): Demanda el aislamiento de datos (IDs unívocos) ante 
     inserciones concurrentes masivas de otros usuarios en el entorno swagger.
   - Restricción de Rollbacks: Carencia de estados transaccionales que obliga a generar 
     mocks persistentes que no son saneados al finalizar la sesión.
