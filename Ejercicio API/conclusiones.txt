HALLAZGOS Y CONCLUSIONES DEL EJERCICIO DE PRUEBAS API PETSTORE
==================================================================

CORRECCIONES IMPLEMENTADAS
---------------------------
1. SINTAXIS NATIVA DE KARATE: Se corrigió el error fundamental de mezclar sintaxis 
   de Cucumber con Karate. Eliminadas las líneas Gherkin decorativas que causaban 
   "no step-definition method match found"

2. ESCENARIOS INDEPENDIENTES Y AUTOCONTENIDOS: Se implementaron 4 scenarios 
   completamente independientes, cada uno con sus propias precondiciones explícitas

3. ESTRUCTURA LIMPIA Y MANTENIBLE: Se organizó el código con secciones claras 
   usando Background para cargar schemas y parámetros comunes

4. ESTILO BDD EXPLÍCITO: Se mantienen pasos Given/When/Then en cada escenario
   para trazabilidad y lectura sencilla

5. IDS ÚNICOS DINÁMICOS: Se implementó generación de IDs únicos usando timestamps 
   para evitar conflictos entre ejecuciones paralelas

6. TAGS PROFESIONALES: Se implementaron tags (@addPet, @getPetById, @updatePet, @findPetsByStatus) 
   para ejecución selectiva de escenarios individuales

7. PATRÓN CREATE-FIRST: Cada test de consulta crea explícitamente los datos 
   necesarios en lugar de asumir que existen

8. VALIDACIONES COMPLETAS: Cada escenario valida entradas, procesa la operación, 
   y verifica salidas según los requisitos del ejercicio

9. CONTRATOS EXTERNOS: Se separaron schemas strict/flex en archivos JSON para
   validación de contrato reutilizable y mantenible

10. FACTORY CENTRALIZADA: La generación de datos y utilidades se centralizó en
    karate-config.js para reutilización estandarizada

RESUMEN
-----------------
Se implementaron exitosamente pruebas automatizadas para la API de PetStore utilizando 
Karate Framework, cubriendo todos los escenarios solicitados: creación, consulta, 
actualización y búsqueda de mascotas. Los tests validan tanto la funcionalidad básica 
como la integridad de los datos a través de todo el flujo de operaciones CRUD.

3. OPERACIONES IMPLEMENTADAS Y VALIDADAS (FUNCIONANDO CORRECTAMENTE)

   📋 COMPORTAMIENTO ESTÁNDAR DE LA API CONFIRMADO:
   La API de PetStore sigue patrones REST estándar:
   - GET a un ID inexistente: Retorna 404 (Pet not found)
   - Los pets deben crearse explícitamente usando POST /pet
   - Los tests implementan el patrón create-first para garantizar disponibilidad de datos

   TEST 1: POST /pet (Añadir mascota) [@addPet]
   - Entrada: JSON con mascota generada dinámicamente con timestamp único
   - Validaciones exitosas: ID único, nombre, categoría, status
   - Comportamiento: Crea la mascota correctamente con datos consistentes
   - Resultado: EXITOSO - Mascota creada con ID único evitando conflictos

   TEST 2: GET /pet/{petId} (Consulta por ID - Patrón create-first) [@getPetById]  
   - Setup: Crea pet usando POST /pet para garantizar disponibilidad
   - Entrada: ID del pet previamente creado en el mismo escenario
   - Validaciones exitosas: Datos del pet creado vs datos consultados
   - Comportamiento: Consulta estándar con validación completa de respuesta
   - Resultado: EXITOSO - Consulta exitosa con patrón create-first

   TEST 3: PUT /pet (Actualizar mascota) [@updatePet]
   - Entrada: JSON con datos modificados generados dinámicamente
   - Validaciones exitosas: Transición de status "available" → "sold" documentada
   - Comportamiento: Actualización completa con preservación de campos no modificados
   - Resultado: EXITOSO - Actualización verificada con auditoria de cambios

   TEST 4: GET /pet/findByStatus (Consulta por status con validación flexible) [@findPetsByStatus]
   - Entrada: Parámetro status="sold" con delays para consistencia eventual
   - Validaciones exitosas: Schema flexible para manejar variabilidad de API demo
   - Comportamiento: Filtrado robusto con validación de todos los pets retornados
   - Resultado: EXITOSO - Búsqueda por status con validaciones adaptables

VENTAJAS DEL FRAMEWORK KARATE
-----------------------------

1. SINTAXIS DECLARATIVA
   - Los tests son legibles y autodocumentados
   - La sintaxis Gherkin facilita la comprensión por parte de no-programadores
   - Separación clara entre configuración, datos y lógica de prueba

2. CAPACIDADES INTEGRADAS
   - Validación nativa de JSON sin librerías adicionales
   - Manejo automático de headers HTTP
   - Generación automática de reportes HTML detallados
   - Soporte nativo para variables y expresiones JavaScript

3. FACILIDAD DE MANTENIMIENTO
   - Un solo archivo .feature contiene todos los escenarios
   - Reutilización de variables entre escenarios
   - Configuración centralizada en karate-config.js

LIMITACIONES ENCONTRADAS
------------------------

1. API PÚBLICA COMPARTIDA
   - Los datos pueden ser modificados por otros usuarios
   - Posibles conflictos de ID si múltiples usuarios usan el mismo
   - Dependencia de la disponibilidad del servicio externo

2. VALIDACIONES DE DATOS
   - La API no valida rigurosamente algunos campos opcionales
   - Permite valores nulos en campos que deberían ser requeridos
   - No hay validación de formato en URLs de imágenes

3. GESTIÓN DE ESTADOS
   - No hay transacciones, los cambios son inmediatos
   - No hay rollback automático en caso de errores parciales
